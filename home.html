"Log Out Everywhere" Is a Lie. Here's What It Would Take to Make It True.

Browser-mediated federated session revocation — a research-grade map of why logout fails across browser state surfaces, and the layered architecture that could actually fix it.

Manishi Gautam · Long-form research document · ~25 min read

"Log out everywhere" is one of the most-promised and least-delivered features in modern web security. Users believe clicking it at Google or Okta terminates their sessions everywhere. In practice, it usually terminates one server record at the identity provider — and leaves every website's client-side state, cookies, stored tokens, service-worker caches, and in-memory credentials fully intact and usable.

This document treats that gap as a real research problem and maps its entire solution space. The central question: can the browser — the one component that physically holds a website's session state — be made into an enforcement point, so a trusted signal can destroy that state even when the website never implemented logout properly?

1 · Logout is not one problem — it's four

Most failed "logout everywhere" promises are a category error: solving one of these problems and claiming the others were solved too. Keeping them separate is the single most important analytical move in this work.

P1 — Local stale stateThe website's client artifacts persist in this browser: cookies, tokens, caches.
BROWSER-SOLVABLE
P2 — Credential validityThe website's server still honors the tokens it issued, regardless of client state.
SERVER-ONLY, BY CONSTRUCTION
P3 — Credential portabilityThe credential was copied off-device before revocation. No cleanup can reach it.
CRYPTOGRAPHIC BINDING ONLY
P4 — Trust & authorizationWho may instruct the browser to destroy state for an origin — scoped and authenticated how?
THE GENUINELY OPEN PROBLEM
The trap inside P4 (P4a): if the identity provider's own login session survives revocation, the website silently re-logs the user in seconds later — and the entire revocation is undone. Any design that skips terminating the provider session is cosmetic, not real.

Plus the supporting cast: P5 multi-account precision (two users, one origin, one storage bucket), P6 offline browsers that never hear the signal, P7 the revocation channel becoming a tracking vector, and P8 the "why was I signed out?" UX cost.

2 · Where sessions actually hide

FACT Two structural facts shape everything. First, Chromium partitions storage by a StorageKey — origin + top-level site + ancestor bit — which contains no "account" dimension. Second, cookies are scoped to the registrable domain (eTLD+1), so clearing abc.com also clears sub.abc.com.

Three load-bearing conclusions INFERENCE:

  1. Every persistent surface is reliably deletable by the browser — but none is labeled by account. Deletion is by origin, never by "which user."
  2. Live memory cannot be scrubbed; it can only be evicted by destroying the execution context — disruptive and reload-based.
  3. Copyable surfaces defeat browser-only revocation entirely once copied. The copy lives outside the browser's jurisdiction.

3 · Why every existing mechanism falls short

Each mechanism below is honest about one problem and silent about the rest. Verdicts: floor (minimum local enforcement), bridge (partial), or real (solves its layer).

Clear-Site-Data — the perfect engine with the wrong trigger

FACT An HTTP header by which a site tells the browser to wipe its own cookies, storage, caches, and execution contexts. Complete destruction — but only the relying party itself can send it, and the entire premise of this research is the non-cooperating relying party. The heart of the proposal is reusing this destruction engine under a different, trusted trigger.

OIDC Front-Channel Logout — eroding into cosmetic

FACT The provider loads each website's logout URL in hidden iframes. The spec itself warns the iframe "might not be able to access the RP's login state" cross-origin — and third-party cookie blocking has largely broken it. Verdict: cosmetic in modern browsers.

OIDC Back-Channel Logout & CAEP/Shared Signals — server-side, client-blind

FACT Signed server-to-server events (CAEP's session-revoked Security Event Token) propagate revocation between cooperating servers in near-real-time. This is the right answer for P2 — but the receiver is a server, never the browser. The device physically holding the client state is absent from the architecture. That blind spot is precisely the gap this research targets.

FedCM — the trust anchor, not yet an enforcer

FACT The browser's FedCM grant ledger records which sites a user federated into through which provider — a privacy-reviewed, browser-witnessed record. It cleanly answers who may revoke what (P4): a provider may act only on relying parties it demonstrably federated. But no IdP-initiated destruction primitive exists today, and the ledger only exists where sites adopted FedCM.

DBSC — the anti-theft layer

FACT Device Bound Session Credentials tie a session to a hardware key (TPM/secure enclave): a copied cookie is worthless off-device, and the server can kill a session by refusing to refresh it. Solves P3 for adopters. It is not a cleanup mechanism and does nothing for sites that haven't adopted it.

Enterprise policy & browser profiles — the honest answers nobody markets

INFERENCE In a managed fleet, trust provisioning solves P4 by fiat, full-origin collateral is acceptable, and extensions can be restricted — the best near-term deployment. And profiles/containers are the only mechanism giving true per-account isolation — by prevention, not deletion.

4 · How revocation gets bypassed

  • Copied tokens (P3). Exported before revocation, they work elsewhere. Browser cleanup is powerless; only DBSC + server revocation help. Bounded only.
  • Tokens already in memory. A live tab keeps making authenticated calls after storage is wiped. Only context destruction evicts them; a sub-second race remains.
  • Service-worker resurrection. A running worker re-creates state post-wipe. Must terminate, unregister, then re-verify nothing re-registered.
  • Re-login resurrection (P4a). The page reloads, the provider session is still alive, and the user is silently signed back in. The single most likely failure in practice.
  • Multi-account collateral (P5). Two accounts, one origin, one StorageKey — the browser cannot target one. Structurally unsolved.
  • Partitioned-storage traps. An origin's state exists under multiple StorageKeys (first-party and embedded). Miss one and the wipe is incomplete — a classic correctness trap.
  • Compromised provider. Equals the provider's existing power to forge identity itself — no new trust assumption; bounded to forced-logout DoS of its own relying parties.

5 · Verdicts — direct answers

Is browser-only logout enough?
No. It's a floor. It reliably destroys local state in one browser and is powerless against server-honored tokens and copied credentials. Claiming otherwise is the central error this document exists to refute.
Is per-account browser deletion possible?
No — structurally. Storage location is a function of (origin, top-level site, ancestor bit). "Account" is not an argument of that function. The only per-account boundaries are browser-owned objects and the profile/container.
Is full-origin deletion the right default?
Yes. The browser can't label data by account or confirm a value is "auth." Destroying everything for the origin is the only model immune to every site's storage-schema quirks.
Can a non-browser-owning IdP (Okta, Facebook) safely order deletion?
Yes — within a strict boundary. Only for relying parties the browser can independently bind to that provider (FedCM grant, enterprise allowlist, or attested binding), with a signature that verifies against its published keys. No binding → no action. Inferring bindings by watching network traffic is privacy-corrosive and lossy; off by default.
What is most likely to fail in practice?
Two front-runners. Flow-inference binding (gutted by per-tenant provider domains; won't survive privacy review) — and any design that skips ending the provider's own session, which demos perfectly and is undone by silent re-login seconds later.

6 · The architecture that could work

No single layer covers all four problems, so the strongest realizable system is five layers, each owning exactly one job and refusing to claim another's:

  • A — Anti-theft (P3): device-bound, short-lived credentials (DBSC/DPoP). Requires site adoption; without it, P3 is simply uncovered.
  • B — Server validity (P2): CAEP session-revoked events to cooperating servers — and the provider always ends its own SSO session (closes P4a).
  • C — Browser cleanup (P1, the novel layer): a browser-process revocation service: verify signal → check binding scope → ordered teardown. No relying-party cooperation needed.
  • D — Enterprise control (P4): policy-pinned trusted providers, mandatory enforcement, managed profiles.
  • E — Recovery (P6/P8): crash-safe journaling, enforce-on-reconnect, provenance UI ("signed out by your identity provider").

The enforcement sequence (Layer C)

On verified, in-scope revocation signal for origin O:

 1. AUTHORITY   revoke FedCM grant · login-status = logged-out
                · clear auto-reauth        (nothing re-logs-in mid-teardown)
 2. LIVE FIRST  terminate service/shared workers · cancel in-flight
                requests · flush sockets   (stop resurrection + races)
 3. PURGE       cookies (ALL partition keys) + local/sessionStorage
                + IndexedDB + caches, across ALL StorageKeys of O
 4. CONTEXTS    tear down live documents · evict bfcache · cancel prerender
 5. VERIFY      confirm no SW re-registered · journal completion

 Block new requests for O until step 3 confirms.
 NEVER touch: saved passwords, other origins, history.
Why the ordering matters: delete the authority to re-authenticate first, kill the live things second, purge disk third. Reverse it, and a service worker or auto-reauth quietly rebuilds the session behind the wipe.

7 · What to prototype and standardize first

Prototype first: the FedCM-grant-scoped revocation path in Chromium behind a flag — signed-event poll → verify → ordered purge — instrumented to measure the two numbers that decide whether this idea is real or cosmetic:

  1. the live-tab race window (how long an in-flight request can still authenticate post-revocation), and
  2. the re-login-within-60-seconds rate (how often a wiped session silently restores).

Standardize first: a FedCM revocation extension — provider revocation_endpoint, grant-scoped enforcement, mandatory logged-out semantics — carrying CAEP's session-revoked event verbatim as payload. One working group, one living spec, inherited privacy review.

Deploy first: the enterprise-managed fleet, where trust provisioning solves the hardest problem (P4) by policy and full-origin collateral is acceptable. The consumer web follows.

8 · Open questions

The measurement gap — the biggest blind spot

INFERENCE No published corpus measures what fraction of top sites are cookie-only vs. localStorage-token vs. service-worker-cached vs. memory-held. That single dataset determines the real-world coverage of any browser floor — and would anchor a strong measurement paper. This comes first.

Account-keyed storage & credential-bound encryption SPECULATIVE

The only two known routes to per-account precision and deletion-free revocation: storage buckets tagged to a federated account (deletes only that account's bucket — but the site must write into them, quietly reintroducing cooperation), and origin storage encrypted under a session-derived key, where revocation destroys the key — fast, offline-capable, copy-resistant ciphertext. Both are far from any engine today.

Governance, latency, cross-engine convergence

Who decides which providers a browser trusts to enforce — user, enterprise, or registry? Does polling suffice or do high-risk cases need push? Will WebKit and Gecko map the same signal to equivalent destruction given different partitioning models?

The bottom line

Today, logout is a courtesy the website may or may not extend. The research agenda is to make client-state termination a property the platform can guarantee on a trusted signal — with the honest caveat, stated plainly and permanently: invalidating a copied credential browser-only is not unsolved. It is impossible, which is a different and more final thing.
