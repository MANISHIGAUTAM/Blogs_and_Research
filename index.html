Featured Feasibility Study

Browser-Mediated Federated Session Revocation

When you click "log out everywhere" at your identity provider, tokens in localStorage, HttpOnly cookies, service-worker caches, and live JS memory survive on every relying party. This study asks: can the browser — which physically holds that state — become the enforcement point?

Verdict: Technically feasible as a Chromium-only prototype today, partially standardizable within 2–3 years — but "perfect" revocation is impossible without sacrificing multi-account convenience and accepting a trust-bootstrapping problem.

Three key findings

  • The deletion machinery already exists. Chromium's Clear-Site-Data / StoragePartition::ClearData can destroy every disk-backed auth surface. The gap is the trusted transport for the revocation signal — and live-tab invalidation of in-memory tokens.
  • Trust bootstrapping is the hardest problem. A browser has no cryptographic relationship authorizing an arbitrary IdP to wipe an arbitrary site. The defensible fix: scope the IdP's authority to RPs the browser observed the user federate into via FedCM — converting "clear any site" into "clear sites you introduced me to."
  • Full origin destruction beats selective cleanup. The browser cannot read auth semantics — it deletes by origin, not by "is this a token." Full StorageKey destruction is the only model that works without RP cooperation, at the cost of logging out all co-resident accounts.

Recommendation

Pursue as a FedCM-scoped, connected-accounts-authorized revocation extension, prototyped behind a Chromium flag + enterprise policy. Target the enterprise / zero-trust market first — where managed trust and multi-account sacrifice are acceptable — before consumer-web standardization. Never market it as "perfect logout": without IdP server-session termination, revocation can be silently undone by reauthentication.

Why today's mechanisms fail

  • OIDC Front-Channel Logout — defeated by third-party cookie blocking; requires the RP to implement it and a live browser session.
  • OIDC Back-Channel Logout — server-to-server only; never touches the browser, so SPA tokens in localStorage and service-worker caches persist.
  • OAuth Token Revocation (RFC 7009) — a courtesy notification, not a guarantee; does nothing client-side and doesn't invalidate issued JWTs.
  • CAEP / Shared Signals + CAE — near-real-time but server-side only, with propagation latency up to ~15 minutes.
The unmet need: enforce revocation even when the RP never implemented proper logout, by making the browser the enforcement point.

Transport & trust model options, ranked

Option A — FedCM-scoped revocation extension BEST

Extend FedCM so an IdP can trigger destruction of local RP state — honored only for RPs in the browser's connected-accounts set for that IdP. Authority is rooted in browser-owned federation records, not IdP pre-registration. Bounded: an IdP can only clear sites it actually introduced. Caveat: only covers FedCM-integrated RPs; legacy OAuth-redirect logins leave no record.

Option B — Signed CAEP Security Event Token ENTERPRISE

IdP emits a signed session-revoked SET over the Shared Signals Framework; the browser (or managed component) verifies against pinned JWKS and maps subjects to RP origins. The only path that handles non-cooperating legacy RPs — because the admin supplies the authorization mapping.

Option C — Clear-Site-Data via RP response SHIPS TODAY

RP emits Clear-Site-Data: "*" on a logout endpoint the IdP can trigger. Zero browser changes — but requires RP participation, failing the core premise.

Options D–G — Push, polling, vendor sync WEAKER

Web-Push-like channel: subscription belongs to the RP's service worker, reintroducing RP dependency. Polling: reliable fallback but latency-bound, with a privacy-sensitive phone-home. Vendor sync/push: only works when the IdP is the browser vendor — not general.

Threat assessment (residual risk after mitigation)

Malicious IdP clears arbitrary sitesBounded

Connected-accounts scoping: IdP can only clear RPs it actually federated.

Spoofed / replayed logout eventsLow

JWT signatures + jti/iat replay cache; re-clearing is idempotent.

Race window during async deletionMedium

Terminate → clear → block ordering bounds it to sub-second; not eliminated.

Compromised IdP mass-revocationMedium

Inherent: the IdP is trusted for its own RPs. Rate limits + visible UI.

Reauthentication-undo (the deepest residual)High

A live IdP session, FedCM auto-reauth, or RP-side passkeys can silently resurrect the session. Browser revocation must be paired with IdP server-session kill — and passkey reauth can't be closed by IdP cooperation at all.

Privacy posture: authority derives from data the browser already holds (no new disclosure), revocation results are never JS-observable, and any polling must be minimized — though credentialed revocation polling is inherently identifying, an unresolved tension.

Incremental Chromium prototype path

Phase 0 — Extension PoC (~weeks)MV3 extension: chrome.browsingData.remove() origin-scoped, triggered by IdP poll. Proves the deletion + transport loop without a Chromium fork.
Phase 1 — Native trigger behind a flagStoragePartition::ClearData + cookie filter, gated strictly on the FedCM connected-accounts set. Enterprise policy controlled.
Phase 2 — Live-tab invalidationStop service workers → destroy/reload renderer documents → flush bfcache → block new origin requests until DataRemovalObserver confirms. (Note: executionContexts teardown must be built — Chromium never shipped it.)
Phase 3 — FedCM trust/transportIdP's credentialed signal sets Login Status → logged-out and triggers cleanup, preventing auto-reauth.
Phase 4 — Enterprise CAEP receiverVerify session-revoked SETs against admin-pinned JWKS; subject → RP-origin mapping via managed config.
Key telemetry: the reauthentication-after-revocation rate. If it exceeds low single digits, the feature is functionally void without mandatory IdP server-session kill.

Unsolved & structural problems

1. The reauthentication-undo problem UNSOLVED

If the IdP server session lives, or the RP supports passkeys, a revoked RP silently re-auths. Pure-browser revocation without server cooperation is incomplete by construction.

2. Live-runtime token clearing PARTIAL

V8-heap tokens can't be surgically removed; only context destruction works — which loses in-flight user work. No non-disruptive solution exists.

3. eTLD+1 cookie blast radius STRUCTURAL

Clearing cookies for one origin clears subdomains too. __Host- prefixed cookies are the partial, RP-cooperative fix.

4. Multi-account disambiguation UNSOLVABLE AT BROWSER LEVEL

Co-resident accounts share one StorageKey; the browser cannot tell user 1 from user 2. Full-origin destruction logs out everyone — acceptable in enterprise, not as a silent consumer default.

5. Bootstrapping for non-FedCM logins SCOPING GAP

Classic OAuth-redirect logins leave no connected-accounts record, so FedCM scoping doesn't apply. These need the enterprise CAEP path or RP cooperation — the original "RP did nothing" premise is only fully met in managed deployments.

6. Cross-browser interop OPEN

Safari/Firefox have different (often more aggressive) storage lifetimes, but no shared revocation API exists. W3C Federated Identity WG is the venue.
