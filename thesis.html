Browser-Mediated Federated Session Revocation · Research Thesis

Can a browser end a session a website refuses to end?

When you click "log out everywhere," the identity provider ends its own session — and the website you logged into keeps treating you as signed in, sometimes forever. This is the complete map of why that happens, what the browser can and cannot do about it, every way the fix can be bypassed, and the architecture that would make logout an enforceable property instead of a website's courtesy.

01 · PROBLEM DECOMPOSITION

"Session revocation" is four problems wearing one coat

Almost every failed "log out everywhere" is a category error — solving one of these and claiming the others came free. Keeping them separate is the single most important move in the whole research.

P1 · Local state
P2 · Server validity
P3 · Portability
P4 · Trust & authority
Residual / impossible
P1 · Browser-owned · solvable

Local stale state

The website's cookies, stored tokens, service-worker caches and in-memory credentials persist in this browser and keep the session usable here — even if the site never wired up logout. This is the part the browser can actually enforce.

P2 · Server-owned · not browser-solvable

Credential validity

The website's backend still honors the token it issued, regardless of client state. No amount of deletion changes what a server chooses to accept. Only the validator can revoke validity.

P3 · Crypto-owned · binding only

Credential portability

The credential may have been copied off-device before revocation, somewhere no cleanup can reach. Only cryptographic binding (DBSC, DPoP) makes a copy worthless. Deletion is powerless here — by construction.

P4 · The open problem

Trust & authority of the signal

Who may order the browser to destroy an origin's state, scoped and authenticated how? Plus P4a: if the IdP's own session survives, the site silently logs the user back in seconds later — undoing everything.

The trap this avoids

"Delete the cookie" solves a sliver of P1 and is routinely sold as if it solved P2 and P3 too. It does not. A complete architecture assigns each problem to a layer and never lets one layer claim another's territory.

First-principles: why logout is many actions, not one

Authenticated state is distributed across independent owners. The browser owns client storage; the relying party (the site you log into, e.g. abc.com) owns its server session and decides what its tokens mean; the identity provider (Google, Okta, Apple…) owns its single-sign-on session and the federation grant. "Logout" can mean deleting a cookie, clearing stored tokens, killing a server session, revoking OAuth tokens, ending the IdP session, tearing down live tabs — or, very commonly, nothing but a UI change while every credential stays valid. No current standard orchestrates all of it.

02 · STATE DISCOVERY

Everywhere a session can hide

You cannot revoke what you cannot find. A session is not "a cookie" — it is a lineage of artifacts scattered across a dozen surfaces, most of which the browser can delete but none of which it can label by account.

Browser CAN destroy reliably

Persistent surfaces

Cookies (incl. HttpOnly), localStorage, IndexedDB, Cache Storage, service-worker registrations & caches, HTTP cache. All deletable by origin / StorageKey — but never by account.

Only by destroying the context

Live surfaces

JS / WASM heap, in-flight requests, running service & shared workers, bfcache, prerender. There is no heap scrub — the only eviction is reload / navigate / kill the tab.

Out of reach

Escaped surfaces

Anything copied off-device, and extension storage. Once bytes leave the browser's jurisdiction, deletion is meaningless. This is the hard ceiling.

The one bright spot

The browser already owns a few per-account objects — passkeys, FedCM grants, DBSC keys, the profile/container boundary. They are the scattered embryos of a session registry that nobody has yet unified.

03 · SESSION IDENTITY

What is a session, really?

A session is a server-side belief, projected onto client artifacts. The server believes "bearer of X is account A until time T." The client merely holds X. Nothing in the web platform reifies that belief as an object. The platform has storage, credentials, and requests — it has no first-class Session.

The true identity of a session

SessionID := ( RP origin , account A , IdP issuer , issuance event e , lineage L )

…where lineage L is the set of artifacts derived from issuance event e: the Set-Cookie that followed the token exchange, the localStorage writes the login page made, the caches it populated, the in-memory copies in live tabs.

Session identity is capturable at exactly one moment — the login event — and trackable afterward only if writes are labeled. The browser can discover it today only for FedCM logins; it can infer it from the OAuth redirect dance and the Set-Cookie that follows; it cannot track or represent it because writes are unlabeled and no Session object exists. Every architecture later in this thesis is a different answer to: how do we capture the moment, and how do we label the lineage?

05 · BYPASS & FAILURE ANALYSIS

The uncomfortable part: copy-and-replay

The question that breaks naïve designs: someone copies the session_id and replays it after you purge. Here is exactly what survives, in each location, and which layer is the only thing that stops it.

Does browser purge stop the replay?
Where the copy is replayedPurge stops it?

Different machine · same browserno
Different machine · different browserno
Token already read into tab memoryno

Stated plainly: once a session_id value is copied anywhere, browser-side purge is powerless against the copy in 4 of 5 location cases, and only partial in the 5th. This is problem P3 — and it is impossible to fix by deletion, because the copy lives outside the deleting browser's jurisdiction.

The only three things that actually defeat a copied id

1 · Make the value worthless when copied

DBSC / DPoP bind the session to a hardware key that never leaves the original machine. A copied id presented from a different browser, machine or profile fails at the server — the only mechanism covering the cross-browser and cross-device cases.

2 · Make the server stop honoring it

CAEP / server-side revocation kills the value at the validation point. Reaches every case, because it doesn't care where the copy physically sits.

3 · Crypto-erase the at-rest copy SPEC

If the stored artifact was encrypted under a session key held in hardware, a copied ciphertext is inert once the key is destroyed. Protects copies of stored data — not a token already read into plaintext.

Why the layering is mandatory, not decorative

This is the proof. Browser purge is a floor. The recommendation was never "purge alone" — it was anti-theft (DBSC) + validity (CAEP) + cleanup (purge), layered, because the bypass table shows purge alone leaving most of the grid red. Go back to the console and toggle the layers: the five cases flip from red to green exactly as this section predicts.

Other realistic failure modes

Service-worker resurrection

A running worker re-creates state or re-registers after a wipe. Fix: terminate running versions + unregister + re-verify no re-registration before any user gesture. Reduced.

In-memory tokens

A live tab holding a token in its heap keeps calling the API after storage is wiped. Fix: destroy the execution context. Reduced; sub-second race remains.

Re-login undo (P4a)

The page reloads and the still-live IdP session silently signs the user back in. Fix: clear auto-reauth and end the IdP session. Mandatory or the whole thing is cosmetic.

Spoofed / replayed signal

Forged or resent instruction. Fix: signature + binding gate + unique-id cache + monotonic serials. Prevented.

Async deletion race

An in-flight request completes before deletion lands. Fix: ordered teardown + block new origin requests until purge confirms. Reduced to sub-second.

Extension reinjection

An extension with host permissions re-injects credentials. Fix: enterprise extension policy only. Bounded — out of web scope.

06 · THE MAPPING PROBLEM

"Where is this site's session_id even stored?"

To purge a session you must first map it. But a value like localStorage["xyz"] = "eyJhbGci…" looks like any other string. The browser cannot read auth semantics. There are exactly three ways to ever build that map — and no fourth.

(a) Inference / observation

Watch & taint

Detect a login event (OAuth redirect returns a code; a credentialed POST gets a Set-Cookie), then taint everything written in the next few navigations as that session's lineage. Works with no cooperation — but lossy, misses lazily-fetched tokens, and can't see into the value to know the account.

(b) Browser owns the moment

FedCM / WebAuthn

When login goes through a browser-mediated API, the browser is the issuance point — it knows RP, IdP, account, timestamp natively. Precise and per-account — but only covers logins that used those APIs (a minority today).

(c) The site tells you

RP declares

The site writes its session into an identity-scoped bucket or declares it via a navigator.sessions-style API. Precise & per-account — but violates the "no RP cooperation" constraint.

There is no fourth option

You cannot get a reliable, per-account session_id → location map for an arbitrary uncooperative site and avoid both inference's lossiness and the in-memory blind spot. A token in Redux state in a live tab is invisible to every method above — you can't enumerate it, only evict it by tearing down the tab.

07 · RECOMMENDED ARCHITECTURE

Five layers, each owning exactly one problem

No silver bullet — a layered system where each layer is independently useful and incrementally adoptable, and none pretends to cover another's job.

Layer A

Anti-theft

Device-bound, short-lived credentials (DBSC; DPoP for token APIs). Makes a copied credential worthless off-device.

requires RP adoption
Layer B

Server validity

IdP emits CAEP session-revoked; RP receivers kill server sessions; the IdP always ends its own session (closes the re-login undo).

IdP-cooperative
Layer C

Browser cleanup · the floor

The novel layer: verify signal → check binding/policy scope → ordered teardown of all client surfaces. No RP cooperation.

browser-only
Layer D

Enterprise control

Policy-pinned trusted IdPs, mandatory enforcement, extension restrictions, managed profiles. Solves trust where it's hardest.

admin-provisioned
Layer E

Fallback & recovery

Crash-safe journaling, enforce-on-reconnect, provenance UI, per-IdP distrust toggle.

resilience
Must never be cleared

Revoking a session must never destroy the password manager / saved credentials — that's the re-login path. Nor other origins' data, profile-wide settings, or history. The floor and the ceiling of acceptable deletion nearly coincide: everything for this origin, nothing outside it.

08 · THE OPTIMAL FLOW

One button, the right order

Order is everything. Suppress re-auth first so nothing logs back in mid-teardown; kill live state before deleting storage so a running worker can't resurrect it; verify last. This is the sequence the console runs.

09 · PRIMITIVES & CRYPTOGRAPHY

The two ideas that change the game

Of twenty primitives the problem needs, the platform has about nine. The missing ones cluster around three themes — capture the login moment, label the lineage, address the browser. Two cryptographic primitives, if built, would make most of the hard problems dissolve.

SPECULATIVE · the jewel

Credential-bound storage encryption

Each session's lineage is encrypted under a SessionKey held in hardware. Revocation = destroy one key. Instant, offline-capable, race-free, per-session precise, and ciphertext copies die with the key. Turns revocation from enumerate-and-delete into forget-one-secret.

SPECULATIVE · solves multi-account

Account-keyed storage

The storage key grows an identity dimension: (origin, top-level site, account). Per-account revocation becomes a native delete. The multi-account problem — proven impossible today — dissolves by construction.

The recursive discovery

Ask repeatedly "what primitive makes the whole problem easier?" and the same answer keeps surfacing: capture the login moment as a first-class event. Every future primitive — session keys, account-keyed storage, a session registry — is downstream of the platform simply knowing that, and as whom, a login happened. That is the missing keystone.

10 · GOVERNANCE & STANDARDIZATION

What's a browser feature, a web standard, an enterprise policy

Standardizing authority over a website's client state is as much a governance problem as a technical one. Get the layering of who decides wrong and it fails review on privacy or RP-autonomy grounds.

Where each piece belongs

Browser feature

The destruction engine, grant-gating, the verifier, the poller. Prototype in one engine, no standard required.

ship first

Web standard

FedCM revocation_endpoint + grant-scoped semantics; CAEP session-revoked as the payload; a "browser receiver" role in Shared Signals.

multi-engine

Enterprise policy

Trust provisioning outside FedCM, mandatory enforcement, extension restrictions. Stays policy permanently.

managed

Vendor-specific

A browser-owning IdP's account→browser channel as a low-latency accelerator. Acceptable, non-interoperable.

optional
Likely to be adopted vs. fail review

▸ LIKELY ADOPTED

CAEP / SSF (already shipping). A FedCM revocation extension — natural fit, active working group, inherited privacy model. The minimum viable path.

▸ LIKELY TO FAIL REVIEW

Spec-level third-party Clear-Site-Data triggers (RP-autonomy objection — keep it internal). Push-based consumer transport (privacy). Any flow-inference binding — a browser silently building a login graph won't survive review. Forced cross-origin tab reloads draw scrutiny as "disruptive."

Incremental adoption roadmap
Phase 0 · weeks

Extension proof-of-concept

Origin-scoped purge driven by a polled, signed IdP endpoint. Validates the deletion + transport loop with no fork. Measures completeness.

Phase 1 · the enterprise product

Native build behind flag + policy

The ordered teardown over the existing browser purge stack, gated on FedCM grants or policy allowlist. This is the honest first deployment — trust is provisioned, collateral is acceptable.

Phase 2 · the consumer standard

FedCM revocation extension

IdP-initiated, grant-scoped revocation carrying CAEP SETs, with mandatory logged-out + auto-reauth clearing. One working group, one living spec.

Phase 3 · the lift to guarantee

DBSC + CAEP coupling

Device binding everywhere and server-side validity turn the browser floor into a real guarantee against copied and server-honored credentials.

Phase 4 · the science

Session registry + crypto-erase storage

Capture every login, label the lineage, and make revocation a key-destruction operation. The long-horizon research that makes per-account precision native.

11 · THE IDEAL

If a browser were redesigned for this

The Session-Native Browser: every login mints a Session object, every write knows its owner, every session has a hardware-custodied key whose destruction is the revocation.

The web never built the Session as an object — and every difficulty in browser-mediated revocation is the shadow of that missing primitive.

12 · VERDICTS

Feasible now · needs cooperation · impossible

Brutally honest boundaries, sorted by what they demand.

Feasible today
  • Origin / StorageKey purge + worker termination + context teardown
  • FedCM grant revocation + auto-reauth suppression (FedCM logins)
  • Profiles / containers as proactive per-account isolation
  • Enterprise: policy-provisioned trust, mandatory enforcement
  • CAEP server-side revocation for cooperating RPs
Needs cooperation
  • Server-side token invalidation (P2) — RP receiver / short TTLs
  • Anti-theft for copied credentials (P3) — RP adopts DBSC
  • Cross-device / cross-browser kill — server revocation or vendor sync
  • Stop silent re-login (P4a) — IdP ends its own session
  • Per-account precision — RP partitions storage, or future account-keyed storage
Impossible / structural
  • Invalidating a credential that already left the browser, browser-only
  • Per-account deletion of black-box RP storage
  • Scrubbing a live JS/WASM heap without destroying the context
  • Zero race window with async multi-process deletion
  • Reaching extension / native copies; guaranteed delivery to an offline browser
Most worth prototyping first

The FedCM-grant-scoped revocation path behind a flag — instrumented for the two numbers that decide whether it's real or cosmetic: the live-tab race window and the re-login-within-60-seconds rate.

Most worth standardizing first

The FedCM revocation extension, with CAEP session-revoked adopted verbatim as the payload. One working group, inherited privacy review.

Most likely to be cosmetic

Front-channel logout iframes (already broken by storage partitioning), and any "logout" that changes UI while leaving credentials valid. Flow-inference binding looks clever and collapses under per-tenant IdP domains and privacy review.

Biggest unsolved problem

Trustworthy acquisition of the (IdP, RP, account) binding with no RP participation — the authority problem. The runner-up, invalidating a copied credential browser-only, isn't unsolved: it's impossible, which is more final.

13 · OPEN PROBLEMS

The research agenda & the blind spots

Biggest blind spots in current literature
  • Nobody has measured where sessions actually live across top sites — the coverage denominator for every claim here.
  • Logout is specified as an RP courtesy, never as an enforceable property.
  • No browser-receiver role exists in CAEP/SSF — the device holding the state is absent from the architecture.
  • The multi-account problem is unaddressed in FedCM, CSD and CAEP alike.
  • Execution-context revocation has uneven cross-engine support and no defined interaction with bfcache/prerender.
  • The re-login undo loop is undocumented as a failure mode in any logout spec.
Next-step experiments
  • Session-storage census — crawl top-N sites; classify cookie-only vs token-in-storage vs SW-cached vs memory-held.
  • Extension PoC measuring deletion completeness across surfaces.
  • Race-window measurement on the ordered teardown.
  • Re-login-rate measurement with and without IdP session termination — the number that separates real from cosmetic.
  • Service-worker-zombie test: verify no worker serves authed responses post-enforcement.
  • Adversarial signal tests: forged, replayed, flooded.

Browser purge is a floor. Device binding is the anti-theft layer. Server evaluation is the authority. Compose them — don't confuse them.
